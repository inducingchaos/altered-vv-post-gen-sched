# MVP Plan

## Goal

Build a one-click workflow that takes a conceptual prompt and turns it into an annotated, animated, scheduled video for an authorized Instagram account.

## Product Slice

The MVP should optimize for one durable path:

1. Authenticated user signs in.
2. User connects one Instagram account.
3. User enters a conceptual prompt.
4. System generates a structured storyboard.
5. System maps the storyboard into a Remotion composition.
6. System renders a publishable video asynchronously.
7. User reviews the result, picks a schedule time, and confirms.
8. System publishes the video through a queued job at the scheduled time.

Everything outside that path should be deferred unless it materially improves reliability.

## Architecture

### App

- Keep Next.js as the host for pages, route handlers, server actions only where useful, and API surfaces.
- Use a features-based structure under `src/domains/`.
- Keep each functional group split into small files with single responsibilities.

Suggested initial layout:

```text
src/
  app/
  domains/
    auth/
    instagram/
    prompts/
    storyboards/
    videos/
    scheduling/
    publishing/
    shared/
  lib/
```

### Data

- Use Neon Postgres with Drizzle ORM.
- Prefix every table with `vvpgs_`.
- Start with explicit workflow tables so retries and auditing remain straightforward.

Suggested initial tables:

- `vvpgs_users`
- `vvpgs_sessions`
- `vvpgs_accounts`
- `vvpgs_instagram_accounts`
- `vvpgs_projects`
- `vvpgs_prompts`
- `vvpgs_storyboards`
- `vvpgs_video_renders`
- `vvpgs_publish_schedules`
- `vvpgs_publish_jobs`
- `vvpgs_job_events`

### Validation and transport

- Use ArkType for request and domain validation.
- Use React Query for client state and async cache management.
- Add tRPC only for domains where typed client-server procedures reduce friction more than simple route handlers.

### Queueing and orchestration

- Use Upstash QStash for async job execution.
- Make each pipeline stage idempotent.
- Persist status transitions and failure reasons at every step.
- Centralize retry policy and result-style error handling from the start.

Initial queued stages:

1. `generate-storyboard`
2. `build-remotion-input`
3. `render-video`
4. `schedule-publish`
5. `publish-instagram`

## Video System

### Visual language

- Monochromatic palette first, with support for a future single accent color.
- Use fractional HSL lightness values for all video color tokens.
- Prefer off-black for dark backgrounds and light-mode foreground contrast.
- Keep the style brutalist and minimal: 2px strokes, no shadows, strong spacing, restrained motion.
- Build on the feel of Visualize Value-style explainer animations without depending on a single composition.

### Template approach

Start with one reusable composition system made of scene primitives:

- title scene
- statement scene
- comparison scene
- sequence scene
- callout scene
- summary scene

Each scene should accept structured content and timing metadata rather than free-form JSX.

## UX Scope

### MVP screens

- sign in
- connect Instagram account
- create project from prompt
- review generated storyboard
- review render status
- schedule publish
- view publish status history

### Non-goals for the first slice

- multi-account publishing
- collaborative editing
- advanced template editor
- multiple output aspect ratios
- analytics beyond basic publish status

## Delivery Plan

### Phase 1: Foundation

- Initialize shared design tokens for the monochrome brutalist theme.
- Set up the `domains` file structure.
- Add Better Auth.
- Add Drizzle with Neon.
- Add ArkType, React Query, and the shared result/logging utilities.
- Replace starter README guidance later so it matches `pnpm` usage and current stack.

### Phase 2: Durable backend spine

- Create schema and migrations for the core workflow tables.
- Add DB access modules and reusable CRUD pipelines.
- Add queue abstraction and QStash handlers.
- Add consistent logging and retry boundaries.

### Phase 3: Video generation

- Install and wire Remotion.
- Define composition config and theme tokens shared with the app.
- Implement the first set of scene primitives.
- Create a deterministic storyboard-to-composition mapper.

### Phase 4: Product workflow

- Build the create-project flow.
- Store prompt input and generated storyboard data.
- Kick off async rendering.
- Surface render and job status in the UI.

### Phase 5: Scheduling and publishing

- Add Instagram account authorization and persistence.
- Create schedule records and delayed publish jobs.
- Implement publish execution and failure reporting.

### Phase 6: Hardening

- Add linting, tests for core mapping logic, and smoke checks for queue handlers.
- Verify idempotency for all async stages.
- Add operator visibility for retries and terminal failures.

## Immediate Next Implementation Chunks

1. Install and configure the base stack: Better Auth, Drizzle, Neon, ArkType, React Query, and the shadcn preset import path.
2. Establish `src/domains/`, shared theme tokens, and core utility patterns.
3. Add the first database schema with `vvpgs_` table prefixes.
4. Wire QStash and define the pipeline contract types.
5. Add the first Remotion composition and storyboard mapper.

## Risks To Manage Early

- Instagram publishing constraints may shape media requirements and auth flow details.
- Remotion render execution strategy must fit the deployment target and queue model.
- The starter app still carries generic template defaults that should be removed once the real stack lands.
- Shared database usage makes table prefix discipline mandatory from the first migration onward.
